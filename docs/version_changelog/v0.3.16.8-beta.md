# VERSION: v0.3.16.8-beta

# Walkthrough — Replace Legacy --link with Shared Docker Network

This version completely resolves the persistent "Error establishing a database connection" in WordPress by replacing the fragile legacy Docker `--link` mechanism with a proper shared Docker bridge network.

## Root Cause

The error `Cannot link to a non running container: /nf-db-wp-db-wordpress` occurred because Docker's `--link` flag is a **legacy feature** that requires the target container to be actively running at the exact moment the source container is created. If the database container was briefly stopped, restarting, or created after the app container, the link would fail and the app container couldn't connect.

## Solution

Replaced `--link` with a **shared custom Docker bridge network** named `nanofly`. This is Docker's modern, recommended approach:

1. **All NanoFly containers** (databases and applications) are connected to the `nanofly` bridge network
2. Docker's built-in DNS automatically resolves container names on custom networks
3. Containers can discover each other **by name** without needing `--link` or host port forwarding
4. The database does **not** need to be running when the app container is created — DNS resolution happens at connection time, not at container creation time

## Changes Made

### 1. Docker Manager (`manager.go`)
- **Added** `EnsureNetwork()` method: creates the `nanofly` bridge network if it doesn't exist
- **Added** `NanoflyNetworkName()` exported getter for the network name
- **Modified** `CreateDB()`: connects new database containers to the `nanofly` network after creation
- **Modified** `DeployApp()`: removed `Links` parameter, connects app containers to the `nanofly` network after creation
- **Removed** the `links []string` parameter from `DeployApp` signature entirely

### 2. Windows Stub (`manager_windows.go`)
- Updated `DeployApp` stub signature to match
- Added `EnsureNetwork()` and `NanoflyNetworkName()` stubs

### 3. Service Deployment (`service.go`)
- **Image deploy**: removed link query block, calls `DeployApp` without links
- **Git deploy (`gitDeploy`)**: replaced `--link nf-db-<name>:<name>` with `--network nanofly`
- **Local deploy (`localDeploy`)**: replaced `--link nf-db-<name>:<name>` with `--network nanofly`
- **`enrichWordPressEnv`**: `WORDPRESS_DB_HOST` now points to `nf-db-<serviceName>:<internalPort>` (the Docker container name) instead of `host.docker.internal:<hostPort>` — this is resolved by Docker DNS on the shared network
- **Port detection**: Added proper internal port detection for PostgreSQL (5432), MongoDB (27017), Redis/KeyDB/Dragonfly (6379), ClickHouse (8123)
