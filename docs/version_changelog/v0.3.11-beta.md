# Walkthrough: Traefik Reverse Proxy Integration (Traefik)

The first step of our enterprise evolution is complete! NanoFly is now powered by **Traefik**, granting you automatic custom domains and free Let's Encrypt SSL certificates.

## What Was Accomplished

### 1. The Traefik Manager
- Created `internal/proxy/traefik.go` which acts as our proxy orchestrator.
- When you start NanoFly, the Go backend automatically checks if a Docker container named `nanofly-traefik` is running.
- If it isn't running, NanoFly automatically spins it up. It binds to **ports 80 and 443** on your host server, mounts the Docker socket so it can listen for container changes, and creates a secure `acme.json` file for your SSL certificates.

### 2. Dynamic Domain Injection
- I updated the deployment pipelines in `internal/api/services/service.go`. 
- Now, whenever you click **Deploy** on an application, NanoFly queries your `domains_v2` database table.
- If you have domains attached to that app, NanoFly dynamically injects Traefik routing labels into the `docker run` command (e.g., `traefik.http.routers.myapp.rule=Host('example.com')`). 
- As soon as the container spins up, Traefik detects these labels instantly, routes the traffic, and requests an SSL certificate!

### 3. User Interface Updates
- **Domain Instructions:** I added a helpful warning in the `ProjectDetail.jsx` Domains tab. It reminds you that because proxy routing works via Docker labels, you must click **Deploy** again after changing a domain for the rules to take effect.
- **System Settings Tab:** I added a brand new **System** tab inside the Settings page (`web/src/pages/Settings.jsx`). This page warns you (or other admins) not to install Nginx or Apache on the server, as NanoFly's Traefik instance strictly requires complete ownership of ports 80 and 443.

## Verification
- Both the Go backend and React frontend built successfully.
- The next time you start the `cmd/nanofly` server, it will automatically download and start the `traefik:v2.10` container!

> [!TIP]
> Ensure your firewall allows inbound traffic on **TCP ports 80 and 443** so Let's Encrypt can successfully verify your domains.



# Walkthrough: Per-Project Backup & Import System 

I have successfully transformed the backup infrastructure from a simple global zip file into a comprehensive **Project-Level Backup & Restore Engine** that supports per-service logic.

## What Was Accomplished

### 1. Persistent Host Volumes
Previously, NanoFly databases stored their data inside ephemeral container layers. 
- I modified the `docker.Manager` (`internal/api/docker/manager.go`) to explicitly map persistent host volumes for databases.
- Now, whenever a database starts, its internal data directory (e.g., `/var/lib/postgresql/data` for Postgres, `/var/lib/mysql` for MySQL/MariaDB) is mapped directly to `~/.nanofly/volumes/db_{service_id}` on your host machine.
- This ensures your data is safe even if a container is deleted or recreated.

### 2. Project-Level Cron Backups
- Modified the SQLite database (`internal/db/db.go`) to add `backup_enabled`, `backup_time`, and `backup_retention` properties directly to the `projects` table.
- Extended the Go backend (`internal/api/projects/handler.go`) with new endpoints to update these backup schedules.
- Added a new `projectBackupScheduler` loop inside `server.go`. Every minute, it checks if any project is scheduled for a backup at that time, and if so, it automatically kicks off the backup routine for all services within that project.

### 3. Logical Dump Engine (Backup & Import)
Instead of just tarballing raw files, the backend can now generate true logical backups:
- **Backup (`POST /api/v1/services/{id}/backup`):** Triggers a `docker exec` command inside the database container. It runs native tools like `pg_dump`, `mysqldump`, or `mongodump` and streams the `.sql` or `.archive` file directly into the persistent host volume.
- **Import (`POST /api/v1/services/{id}/import`):** Accepts the name of a dumped file. It executes the corresponding restore command (`psql`, `mysql`, `mongorestore`) to wipe the current tables and restore the data from the snapshot.

> [!TIP]
> The backend architecture is fully complete and compiled successfully! (I also fixed a small compilation issue for Linux environments that wasn't previously visible on Windows). The API routes are ready and wired into the React Frontend UI. 

### 4. Frontend Backup UI 
- Modified `web/src/api/client.js` to include the `backup` and `importBackup` methods.
- Added a `BackupRestorePanel` to `web/src/pages/ProjectDetail.jsx` specifically for database services.
- This UI provides a **"Manual Logical Backup"** button and an **"Import Backup"** section that accepts an uploaded `.sql` or `.archive` filename. It dynamically appears alongside your connection details only when you click on a database.

## Next Steps
Everything compiles correctly across both Go (Windows/Linux architectures) and the Vite/React frontend! You can launch the application and navigate to any Database resource within a project to try generating and importing backups on the fly.




# Newly added Support for GitHub Apps Walkthrough


I've successfully implemented the GitHub App integration! This setup mimics the seamless "Coolify-style" one-click experience you requested for securely handling private repositories without manually generating and copying tokens every time.

Here's an overview of what was built:

1.  **"One-Click" GitHub Setup**: Added a new **"GitHub Apps"** tab inside the Settings page. You can simply enter an app name (e.g., `nanofly-production`) and click a button. NanoFly automatically crafts a GitHub manifest and securely redirects you to GitHub to register the App with the required permissions.
2.  **Automated Handshake**: We added callback endpoints that securely receive the generated credentials (including the `pem` private key) from GitHub as soon as the app is created and immediately saves them to your database.
3.  **No More PATs**: When you deploy a private repository, NanoFly's backend dynamically generates a short-lived **Installation Access Token** using JWTs (signed with the App's private key) and uses it to perform the `git clone`, dropping the need for manual Personal Access Tokens entirely!
4.  **Automated Webhooks**: Because your GitHub App inherently routes all events to the URL we defined in the manifest, I set up a global webhook handler at `/api/webhooks/github`. Any pushes to an authorized repository automatically trigger NanoFly to fetch the latest code and seamlessly redeploy the app.
5.  **Easy UI Selection**: When creating a new Private Repository app inside your projects, there's a new "Select GitHub App" dropdown where you can quickly select your connected app from the settings instead of pasting a token.
