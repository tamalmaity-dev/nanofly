# VERSION: v0.3.18

# systematic Container Naming, WordPress DB Resolution, Traefik HTTPS, and Stack UI Grouping

## Changes

### 1. Systematic Production Suffix-based Container Naming
- **Fix**: Replaced all remaining hardcoded container string lookups, stops, statistics mapping, and logical database backups/imports in `service.go` and `backup.go` with structural `svc.ContainerName()` helper method.
- **Benefit**: Ensures all container lifecycle and backup operations consistently reference the production-ready naming scheme (`nf-[db|app]-<name>-<serviceID[:8]>`) which prevents name conflicts on redeployment.

### 2. WordPress Database DNS Resolution on Shared Bridge
- **Fix**: Updated `enrichWordPressEnv` in `service.go` to scan the database service `id` and configure `WORDPRESS_DB_HOST` and `DATABASE_URL` with the correct suffix-based host name (`nf-db-<dbName>-<dbID[:8]>`). This allows WordPress to resolve and connect to its database container over the shared Docker bridge network.
- **Linked Database Checking**: Updated database checks in `Deploy` to look up the DB container with its correct suffix, preventing name conflicts with running databases.

### 3. Traefik HTTPS Asset URLs
- **Fix**: Configured the site URL in `enrichWordPressEnv` to use the `https://` protocol when domain mappings are present. This forces WordPress to generate secure asset links under Traefik's proxy, resolving the unstyled plain-HTML/mixed-content blocking issue.

### 4. Coolify-Style Stack UI Grouping
- **Component Design**: Designed the new `StackCard` component in `ProjectDetail.jsx` which visually groups a WordPress application and its linked MySQL/MariaDB database into a single, cohesive stack card.
- **Grouping Logic**: Configured the project details page to compute stack associations and show a new **Service Stacks** section alongside standard **Applications** and **Databases** categories, with seamless direct navigation to respective tabs.
