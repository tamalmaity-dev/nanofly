# NanoFly v0.3.20.1-beta — WordPress Database & Lookup Improvements

**Release Date:** 2026-05-28  
**Type:** Minor Release / Feature & Bug Fixes  
**Urgency:** Medium-High  

---

## Summary

This release resolves various edge cases around WordPress and linked database setups. It fixes stuck WordPress deployment states, prevents SQLite database duplication, makes automatically-created databases naming scheme more clear, and introduces a robust port-based lookup strategy to connect applications with correct database instances.

## Changes & Fixes

### 1. Robust Database Lookup Heuristics by Port
* **Problem**: Previously, linked databases were resolved solely by name (`wp-db-${svc.Name}`). If a custom database was linked or if names collided, the backend could not identify the correct database container.
* **Solution**: NanoFly now extracts the port number from the saved `WORDPRESS_DB_HOST` environment variable (e.g. `32890` from `host.docker.internal:32890`) and resolves the database service uniquely by that port number first. It falls back to name conventions (`wp-db-${svc.Name}`, `${svc.Name}-mysql`, `${svc.Name}-mariadb`) and generic matches if lookup by port fails.
* **Files Modified**: 
  - `internal/api/services/service.go` (`Deploy()`, `enrichWordPressEnv()`)

### 2. Descriptive DB Service and Container Names
* **Problem**: Database container names were hardcoded with a generic `wp-db-` prefix, hiding the engine type (MySQL vs. MariaDB).
* **Solution**: Automatically-created databases are now explicitly named `${appName}-mysql` or `${appName}-mariadb`, resulting in clear container names like `nf-db-wordpress-mysql-xxxxxxxx`.
* **Files Modified**:
  - `web/src/pages/ProjectDetail.jsx`
  - `internal/api/services/service.go`

### 3. Duplicate Database Service Prevention
* **Problem**: Duplicate database records could be created in the `services` table if previous failed deployments left behind database rows with the same name.
* **Solution**: The frontend now checks if a database service with the same name already exists in the project and cleans it up via the API before creating a new database service.
* **Files Modified**:
  - `web/src/pages/ProjectDetail.jsx`

### 4. Database Persistence
* **Problem**: Automatically deleting the database when deleting WordPress could lead to data loss if users wished to keep their database.
* **Solution**: Deleting the WordPress app service no longer deletes the database service. The database service stays alive and can be deleted separately or linked to a new app.
* **Files Modified**:
  - `internal/api/services/service.go` (`Delete()`)

### 5. Fail-Fast Readiness Checks
* **Problem**: If the MySQL/MariaDB database container crashed or exited during initialization, the WordPress deployment loop would get stuck waiting for `"ready for connections"` for a full 60 seconds.
* **Solution**: The loop now checks if the container is still running. If the container exits or crashes, it fails fast, breaks the loop immediately, and logs the error.
* **Files Modified**:
  - `internal/api/services/service.go` (`Deploy()`)

### 6. Resource Limits Alert Notice for Databases
* **Problem**: Database containers could easily crash with OOM errors if set to small tiers like `micro` or `nano`.
* **Solution**: Added a warning notice in the resource limits panel warning users that databases default to `unlimited` to prevent OOM crashes, urging caution if changing it.
* **Files Modified**:
  - `web/src/pages/ProjectDetail.jsx`

### 7. Fixed JSX Nesting Syntax Error
* **Problem**: A missing closing `</div>` tag in the `ResourceLimitsPanel` broke the JSX tree causing Vite build failures.
* **Solution**: Added the missing closing `</div>` and verified Vite production build.
* **Files Modified**:
  - `web/src/pages/ProjectDetail.jsx`
